# 🚨 高考志愿填报数据 OCR 解析核心错误剖析与重构指南

**当前数据状态评估：** 
经对比原始图片与 OCR 提取数据，当前提取结果存在**极其严重的漏识别、错位和截断问题**。此类数据如果直接入库供学生填报志愿使用，将导致极其严重的退档事故（如丢失体检限报要求、计划数错乱、专业丢失）。

以下是导致数据异常的五大核心根源及重构方向：

---

## ❌ 错误一：括号解析逻辑彻底崩塌（最危险错误）

当前代码在处理“专业名称”与“专业备注”的拆分时，逻辑过于简单粗暴（通常是遇到第一个括号就截断），导致官方极其重要的限制性备注全部丢失。

*   **案例对比：0056 西南大学 (物理类 101组) 91 专业**
    *   **原图数据**：`91 学前教育(国家公费师范生)(专业备注:不招色弱、色盲考生。)`
    *   **当前输出**：`专业名称: 学前教育` | `专业备注: 国家公费师范生`
*   **严重后果**：真正的体检限制要求（不招色弱色盲）丢失！若患有色盲的考生据此填报，投档后将被直接退档。
*   **同类受害者**：0060 陕师大 35专业（丢失雁塔校区、单色识别不全者限制）；所有带有 `[V]` 标记的医学类专业。

> **❓ 核心反思与改进提问：** 
> 你的代码在拆分字段时，是不是用了类似 `split("(")[1]` 或非贪婪正则 `\((.*?)\)`？
> **建议**：绝不能盲目切分第一个括号。应专门定位关键字 `(专业备注:` 或 `(包含专业:` 进行靶向切割。

---

## ❌ 错误二：折行与跨页导致的大规模数据蒸发

当文档排版出现多行折行，或者一个专业组跨越两张图片时，当前的“逐行读取”逻辑无法做到状态延续，直接导致大批专业人间蒸发。

*   **案例对比 1：跨页丢失（图4底 -> 图5顶）**
    *   **原图数据**：5122 西华师范大学 169组 在图4底部，其下属专业（BL, 9U, 47, AJ, AV, 94）在图5顶部。
    *   **当前输出**：西华师大的物理类只提取了 131和143组，**169组及之后的所有专业全部消失！**
*   **案例对比 2：多行折行丢失（图6）**
    *   **原图数据**：0012 北京语言大学 47专业，其名称和备注长达 4 行。
    *   **当前输出**：仅保留了第一行的 `外国语言文学类` 和 `小语种`，后 3 行（包含专业详情、外语口试要求）全部丢失。

> **❓ 核心反思与改进提问：** 
> 当处理新的一页或新的一行时，程序是否丢弃了上文的“上下文状态”（当前院校、当前专业组）？
> **建议**：必须引入**“块累加器 (Block Accumulation)”**机制，只要没有遇到新的专业代码，就必须将当前行的文本拼接到上一个专业中。

---

## ❌ 错误三：数字错位串行（数学校验机制形同虚设）

即使成功提取了行，右侧悬浮的“计划数”也发生了严重的张冠李戴，且程序未能拦截。

*   **案例对比：0056 西南大学 (物理类 102组)**
    *   **原图数据**：组计划 = 12。其中 `C1 化学` 计划数为 **6**，`L1 计算机` 计划数为 **6**。
    *   **当前输出**：`C1 化学` 的计划数变成了 **9**（极可能是从上一组错位抓取过来的）。
*   **严重后果**：计划数错误会严重误导考生的报考决策。

> **❓ 核心反思与改进提问：** 
> 为什么 0056的102组，提取出的计划数之和（9+6=15）不等于标称的组计划（12），但程序没有报错拦截？
> **建议**：必须强制启用 `组计划 == SUM(该组各专业计划数)` 的强校验。一旦对账失败，该组数据打回人工复核，绝不能直接入库！

---

## ❌ 错误四：全局状态（历史/物理）跨页串台

在跨图解析时，维护“历史类”和“物理类”的全局状态机出现了混乱。

*   **案例对比：5015 四川外国语大学**
    *   **原图数据**：图2底~图3顶为**历史类**（专业11-25）；图6底为**物理类**（专业19,21-25）。
    *   **当前输出**：历史类漏了跨图的 25 专业；物理类**只提取了 25**，彻底漏掉了 19, 21, 22, 23, 24。

> **❓ 核心反思与改进提问：** 
> 当“二、物理类”这个标题出现在图片中部时，代码是否能立刻捕获并将全局变量 `exam_type` 从历史切换为物理？
> **建议**：每一行文本处理前，优先正则匹配大类标题，实时更新全局 Context。

---

## ❌ 错误五：特殊字符的纯视觉形近字错误

这是纯粹的 OCR 引擎视觉识别错误，但也暴露出后期清洗机制的缺失。

*   **案例对比：5120 四川师范大学 (305组)**
    *   **原图数据**：`4E 化学(南充市阆中市)`
    *   **当前输出**：`化学(南充市闻中市)` （“阆 lang” 被识别成了 “闻 wen”）。

> **❓ 核心反思与改进提问：** 
> 针对这种 OCR 形近字错误，如何在入库前进行修正？
> **建议**：引入字典校验。将提取出的地名与四川省标准的“县市区行政区划字典”进行匹配校验，发现未登录词汇自动报警。

---

## 🚀 终极重构方案：放弃逐行正则，改用“状态机 + 块累加”

千万不要试图通过写更复杂的正则表达式来解决折行问题，那是一条死胡同。请按照以下伪代码逻辑重构核心解析器：

```python
# 初始化全局状态
current_exam_type = "未校验"
current_school = None
current_group = None
current_major_block = None # 核心：专业区块累加器

for line in ocr_lines:
    # 1. 拦截大类切换
    if "一、历史类" in line.text:
        current_exam_type = "历史类"
        continue
    if "二、物理类" in line.text:
        current_exam_type = "物理类"
        continue

    # 2. 拦截院校与专业组 (更新 current_school, current_group)
    if is_school_line(line): ...
    if is_group_line(line): ...

    # 3. 核心：处理专业行与折行累加
    if is_new_major_code(line.left_text): 
        # 遇到新的专业代码（如 13, C1, 4E），说明上一个专业彻底结束了
        if current_major_block:
            # 【执行校验、拆分名称与备注、入库操作】
            process_and_save(current_major_block) 
        
        # 开启新的专业区块
        current_major_block = {
            "text": line.left_text, 
            "plan": extract_plan(line.right_text),
            "fee": extract_fee(line.right_text)
        }
    else:
        # 当不属于特殊标题、学校、专业组、新专业时，统统视为折行文本！
        if current_major_block:
            # 无脑拼接到上一个专业的文本中！
            current_major_block["text"] += line.left_text
            # 如果折行处右侧有遗漏的计划数，补充进来
            if not current_major_block["plan"] and line.right_text:
                current_major_block["plan"] = extract_plan(line.right_text)

# 循环结束后，不要忘记处理最后一个残留在内存中的专业
if current_major_block:
    process_and_save(current_major_block)
```

**重构后的数据处理流：**
文本无损累加完整 ➡️ 定位 `(专业备注:` 精准切割 ➡️ 触发组内计划数 SUM 校验 ➡️ 校验通过后入库。
